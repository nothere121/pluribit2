How pluriƀit Works

Pure peer-to-peer network

pluriƀit nodes connect directly to each other over libp2p. No servers, no hosted RPC endpoints, no trusted infrastructure. Each node independently validates the chain and participates in gossip.

libp2p networking

Nodes use gossipsub, DHT peer discovery, NAT traversal, and relays:

  • gossipsub for blocks and transactions
  • WebRTC / TCP / WebSocket transports (depending on environment)
  • Kademlia DHT for peer discovery
  • Dandelion stem phase for transaction origin privacy

Network participation is decentralized, permissionless, and censorship-resistant.

Consensus: VRF + VDF + GHOST

pluriƀit uses a fairness-driven consensus design:

  • VRF — Verifiable Random Function fairly selects block producers
  • VDF — Verifiable Delay Function enforces real-time and prevents grinding
  • GHOST rule chooses the chain with most accumulated work

No mining arms race. No staking cartel. Fair, time-based block production.

Mimblewimble privacy

pluriƀit adopts Mimblewimble-style private transfers:

  • Pedersen commitments hide amounts
  • Bulletproofs ensure validity without revealing values
  • Stealth addresses and ephemeral keys provide unlinkable payments
  • Cut-through prunes spent outputs for chain efficiency

Balances and transaction history remain private by default.

Kernel concatenation

Instead of aggregating kernels like Grin, pluriƀit concatenates kernels:

  • Supports cut-through optimization
  • Maintains explicit kernel history
  • Preserves full supply auditability

Deterministic supply schedule

Rewards follow predictable halving cycles and supply epochs. Anyone can verify the total supply from local state — no trusted party and no opacity.

Deterministic Protobuf encoding

Blocks, transactions, and p2p messages use deterministic Protobuf serialization, ensuring identical behavior and consensus across environments and platforms.
